# Database Architecture

```text
Services
    ↓
Repositories
    ↓
Drizzle ORM
    ↓
SQLite
```

```text
lib/db/
  index.ts
  schema/
  migrations/
  seeds/
```

Rules:

* Repositories own database access.
* Services never query database directly.
* All writes must be transactional.
* Automatic migrations on startup.
* Offline-first by default.

# Backend Architecture

RetailX uses a local backend inside the same application.

```text
Next.js UI
      ↓
Feature Services
      ↓
Repositories
      ↓
SQLite Database
      ↓
Tauri Native Layer
```

Application responsibilities:

### Frontend

* Pages
* Components
* Forms
* State Management
* Localization
* Navigation

### Backend

* Business Rules
* Database Access
* Transactions
* Reports Calculations
* Printing Logic
* Backup Logic
* Scanner Integrations

The backend lives inside the same codebase and is organized as:

```text
src/

backend/
  modules/
  services/
  repositories/
  db/
  jobs/
  types/
  utils/
```

Example:

```text
backend/
  modules/
    inventory/
    suppliers/
    purchases/
    sales/
    expenses/
    reports/

  services/
    printer/
    scanner/
    backup/

  repositories/
  db/
  jobs/
```

Rules:

* UI cannot access SQLite directly.
* UI cannot contain business rules.
* Business rules live in backend modules.
* Repositories are backend-only.
* Database transactions are backend-only.

# Dependency Direction

```text
Components
      ↓
Feature Hooks
      ↓
Feature Services
      ↓
Repositories
      ↓
Database
      ↓
Tauri Native APIs
```

Dependencies can only move downward.

No layer may depend on a higher layer.

This rule is mandatory for every module added to RetailX.
